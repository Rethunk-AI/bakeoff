No, you misunderstood my intent on model_sources timestamp.  The timestamp should be a new field in the database schema probably just named "updated", but the actual stats themselves should be inside the JSON blob field.  That "updated" timestamp should reflect when all data in the source_metadata JSON blob was sourced, since it should all be pulled basically at the same time.  We can consider that model_hash is also based on the same timestamp, even though it may be computed later because the model data was acquired at that time, or can be traced back to have existed in that state at that time.

If we need to make multiple data queries from the source site to fill out all fields, they should all be acquired within a 1 minute window, and the timestamp should reflect the earliest returned timestamp of the queries.  If they take longer than the 1 minute window, the returned data can be treated as if it were within the window if querying all data a second time returns the same values as the first. If the second time returns any different values, the original round of data is removed and we try again from this new data set.  All queries must be completed within 5 minutes total.

There is no need for a `stats` sub-object in the source_metadata field, as there should be plenty of unique keys available in the json object.
