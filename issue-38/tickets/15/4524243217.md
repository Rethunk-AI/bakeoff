schema_version format: Agreed to simple integer increment for version numbers

Disk structure: proposed schema makes sense and seems easy to manage, audit, and update.

data contract definition:
- models/<uuid>.json - this should probably contain all persisted data from models DB table, and model_sources table.  possibly others.  we will need to discuss this in more detail.
- run_queue/<uuid>.json - you asked `what constitutes a "self-contained job submission"?` but honestly I'm not sure how this relates to the run_queue table.  The output results file from when a model is tested with a prompt should be self-contained for sure.

I think the correct approach to what you are looking to discover here is to list out all the DB tables and determine how each of them is stored on disk:
- first, is it stored on disk?
- if so, is it stored in its own file, or embedded into another; for example model_sources records would be embedded into models disk files, since they have a 1-to-many relationship with the model they are a source for.
- if it is stored in its own file, what is the location of that file relative to the BAKEOFF_DATA_DIR?
- if it is embedded into another table's disk file, which one? and potentially under what key, but you could probably quite easily infer that and I can just object if there is an issue or approve if it is acceptable as is.

Agreed on defining both disk and DB schemas before writing handlers, although I feel both schemas should be very similar.  There should only be minimal divergences between them.  At minimum, fields with text or numeric data should probably be identical in both name and value in almost all cases.  FKs and IDs might be a bit more divergent depending on how things are stored or embedded.  Also, some fields might be dropped in one schema or another, but that is also the minority case.
