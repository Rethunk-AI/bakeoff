**Bastion response — 210001ZMAY26**

@gissf1 — agreed on tracking source statistics. Design proposal:

**Schema addition to `model_sources`:**

```
model_sources
  ...existing fields...
  source_stats       JSONB NULLABLE         -- point-in-time stats snapshot
  stats_fetched_at   TIMESTAMPTZ NULLABLE   -- when snapshot was last taken
```

`source_stats` JSONB rather than typed columns because stat types differ by source:

| source_type | typical stats |
|---|---|
| huggingface | `{"downloads": 50000, "likes": 1200}` |
| ollama | `{"pulls": 3000}` |
| direct_url | `{}` (no platform stats available) |
| local_file | `{}` |

Typed columns would leave most fields NULL for most sources. JSONB keeps the schema clean and accommodates future stat types without migrations.

**Confidence interpretation:**
- Stats are a soft signal only — popularity is not a proxy for safety or correctness, but high download counts do correlate with broader community vetting.
- Stats decay in relevance: `stats_fetched_at` lets the system surface stale snapshots (e.g., flag if >30 days old) and prompt a refresh.
- A future "confidence tier" view can incorporate stats alongside `model_hash` verification status and source curation level.

**Refresh pattern:** a background job (or manual trigger) fetches current stats from each source's API and upserts `source_stats` + `stats_fetched_at`. No new tables required.

— Bastion
