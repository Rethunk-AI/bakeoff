**Bastion response — 210726ZMAY26**

@gissf1 — agreed. Merging stats into `source_metadata` reduces schema surface with no meaningful capability loss at current scale.

**Revised `model_sources` design:**

Single `source_metadata` JSONB field — provider identity and stats co-located under one blob:

```json
{
  "ollama_tag": "llama3:8b-q4_K_M",
  "ollama_digest": "sha256:...",
  "stats": {
    "pulls": 3000
  },
  "stats_fetched_at": "2026-05-21T07:30:00Z"
}
```

**Trade-off acknowledged:** `stats_fetched_at` moving into JSONB means staleness queries become path expressions:

```sql
WHERE (source_metadata->>'stats_fetched_at')::TIMESTAMPTZ < NOW() - INTERVAL '30 days'
```

Rather than a native indexed column. For this system at current scale, JSONB path queries are fast enough. An expression index can be added later if bulk-refresh scheduling becomes a bottleneck.

**Schema delta (revised from prior proposal):**

No new columns added to `model_sources`. `source_metadata` structure extended by convention to carry a `stats` sub-object and `stats_fetched_at` timestamp string. No migration required — new keys populated on first stats refresh.

**bakeoff#14 design settled:**
- `ollama` added to `source_types`
- `source_metadata` = single JSONB, provider identity + stats co-located
- `model_hash` remains deduplication ground truth

Ready for implementation sub-issues when directed.
