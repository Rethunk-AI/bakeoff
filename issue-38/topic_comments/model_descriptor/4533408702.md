**Bastion response — 251008ZMAY26**

@gissf1 — two points, both confirmed.

---

**Version change → new hash → new UUID + predecessor tracking**

Correct. A new version of a model with changed weights produces a different weight file → different SHA256 → different `UUID5(BAKEOFF_MODEL_NAMESPACE, new_hash)`. The old and new are distinct model records; the new record's `predecessor_model_id` field carries the old UUID.

Updated version-change scenario:

1. Ingestion detects multiple hard field diffs + changed `version` field
2. Pipeline surfaces warning: "This looks like a new model version. Create a new disk file for the new version."
3. Operator creates new stub or full disk file (see below)
4. At hash computation time, pipeline automatically sets `predecessor_model_id` to the old record's UUID (if the old record exists in DB by the time the new weights are ingested)
5. If the operator knows the predecessor at stub creation, they can set `predecessor_uuid` in the stub to short-circuit the lookup

This is already fully supported by the schema — `predecessor_model_id UUID FK` is in place. No schema change needed.

---

**Human-created stub files — minimal format**

Agreed. Contributors should not need to compute SHA256 hashes or run weight-inspection tooling to register a model. A minimal stub that the ingestion pipeline can resolve is sufficient.

Proposed stub format (`models/<any-name>.json`):

```json
{
  "schema_version": 1,
  "stub": true,
  "source_url": "https://huggingface.co/microsoft/Phi-3-mini-4k-instruct",
  "name": "Phi-3 Mini 4K Instruct",
  "predecessor_uuid": null
}
```

Only `schema_version`, `stub: true`, and `source_url` are required. `name` is optional but encouraged. `predecessor_uuid` is optional.

Ingestion pipeline stub handling:
- Detects `stub: true` (or absence of `model_id`)
- Resolves URL → fetches model card metadata (params, architecture, quantization, context length, creator, file format)
- Downloads weights file, computes SHA256
- Generates `model_id = UUID5(BAKEOFF_MODEL_NAMESPACE, sha256)`
- Populates all fields, removes `stub: true`, renames file to `<model_id>.json`
- If `predecessor_uuid` present, validates it exists in DB before setting FK

File naming for stubs: any name is acceptable (e.g., `phi3-mini.json`). The pipeline renames to `<uuid>.json` as the final step, so human-chosen names are transient. A `models/pending/` directory could isolate stubs from settled records if naming collisions become a concern — will note this as a design option when the ingestion pipeline is specced.

---

**Summary of updates to conflict/ingest policy:**

| Scenario | Updated behavior |
|----------|-----------------|
| Version change with new weights | Detect new hash → new UUID; auto-set `predecessor_model_id` from old record; surface prompt to operator |
| Human stub (source_url only) | Accepted; pipeline resolves all fields, renames file, clears `stub` flag |
| Stub with `predecessor_uuid` | Validated against DB before FK is set |
