As initially mentioned in #13:

Model descriptor file format - I agree JSON makes sense, I think versioning makes sense too, so we can import older models and other DB data as needed.

Per your suggestion, `schema_version` is a reasonable field name for versioning.  I imagine this could be just a "schema_version: 1.0" in the file, or similar.  It might make sense to use a date-based versioning scheme, but that is just one option, not a hard opinion.  An example of a date-based versioning scheme would be using formatted dates like YYYYMMDDNN where YYYY is the 4 digit year, MM is the 2 digit month (with zero prefix when less than 10), and DD is the 2 digit day of the month (again, with zero prefix when less than 10), and NN is a serial index where "00" for most cases, but if there happened to be multiple releases in a day, one could increment this to keep them unique and reset it to zero with the next update on another day.  The likelihood of more than 100 updates on a single day seems quite unlikely in normal development scenarios, but in the case we did encounter it, we can address it then.  This format also provides some level of validation that a "known version" is used.  That said using simple incrementing numbers (like 1.0 or 1) are simpler to remember.  I'm open to discussing this more.

Exact location on disk will obviously depend on where we are being run from.  We should not assume the model is in the same directory as the JSON model file data.  in fact, I would think they should be in different directories since the model weights (the model source file) may be cached in some system-wide location whereas the model file data should be managed by us in our file-data locations.  The model data will need to persist beyond the queue usage, so it should be in different directory than the run_queue files.  I think it might make sense to have a "persistent data store" where we can structure the filesystem arrangement based on the intended database structure - using directories for tables, files for records, etc.  Since we need to assume that the runner may not have access to the database, all data should be accessible through file exchange.  I would say this is closer to option B, but not exactly.  the run_queue files should not be in the same directory as the disk-backed table data files (like the model disk file).  In fact, I would say that a run_queue disk file should be relatively self contained since it is a job submission request, and may need to populate other disk files (such as model disk files) as the queuing system prepares to do testing.  The files on disk likely should use their UUID as the filename for the record they contain.  Similarly, there may be a directory for each table (named the same as the table), under some base path.  The run_queue itself is a bit of a special case here, since the data is much less static, so maybe it can be an exception.  Let's discuss the disk structure in more detail for various types of files we will be using.

Q3: model metadata originates from the model_source.  it is stored in the models database table, and the models disk files that back them.  new models are likely going to be submitted as a URL for a known source (huggingface URL, ollama source, etc; but known and relatively trusted sites, not just random untrusted URLs).  We would then have a handler than queries the source for details and determines model data/metadata and how to queue it for testing.  It would also take that acquired data, validate it, and store it in the database and the model disk file backing the database entry.  Specifically regarding your options:
- I don't think manual submission of model.json files is going to be acceptable, other than maybe by administrators or for our initial seeding.
- I don't think CI is the right place to do this since the data can be large and require internet access.
- I don't think we want full auto-discovery of new models, but auto-discovery and validation of the data is the goal here.  We should be able to get the data from huggingface (or other sources) without pulling the full model initially.  Once we do pull the model weights, we should validate the embedded metadata matches and generate the model_hash at the same time.  That said, the verification of the embedded metadata is likely a more complicated task (even compared to generating the model_hash), so I'm okay with putting that embedded metadata verification off for a bit.
