**schema_version format:** Recommend simple integer increment (`schema_version: 1`, `2`, etc.) over YYYYMMDDNN. Date-based versioning couples the schema to a calendar, adds parsing complexity, and provides no validation advantage a schema registry can't give more cleanly. YYYYMMDDNN is a good release artifact naming convention — not a format version identifier. If we need human-readable provenance in the file, add a separate `schema_updated` timestamp field alongside the integer version.

**Disk structure:** The directory-per-table, UUID-filename pattern is confirmed. Proposed base layout:

```
<BAKEOFF_DATA_DIR>/
  models/
    <uuid>.json          ← persistent model record
  run_queue/
    <uuid>.json          ← ephemeral job submission
  (future tables as subdirs)
```

`BAKEOFF_DATA_DIR` env-configurable, no hardcoded path. run_queue as ephemeral exception is correct — job files are self-contained submissions, lifecycle (created → consumed → archived/deleted) differs fundamentally from persistent model records. Agree they should not share a directory.

**Model metadata sourcing:** Agreed on all three negatives — no manual JSON submissions (except admin/seed), no CI, no full auto-discovery. The URL-submit → upstream metadata pull → validate → store (DB + disk file) path is the right architecture. Pulling metadata without weights first is essential for cost and latency; generating `model_hash` at weight-pull time is the correct coupling point.

Deferred embedded metadata verification is accepted. Flag it as a P2 task so it doesn't fall off the backlog.

**Next gate — data contract definition:** Before implementing handlers, I'd like to align on the exact field sets for the two core disk file types:

1. **`models/<uuid>.json`** — what fields does a persisted model record carry? At minimum: `schema_version`, `uuid`, `source_url`, `source_type` (huggingface/ollama/etc), `model_hash` (nullable until weights pulled), `metadata` blob from upstream, `created_at`, `status`. What else?

2. **`run_queue/<uuid>.json`** — what constitutes a "self-contained job submission"? Candidate fields: `schema_version`, `uuid`, `model_uuid` (or `source_url` if model not yet registered), `requested_at`, `requester`, `status`, `result_ref`. What's the minimum viable contract?

Proposing we define both schemas explicitly before writing handlers — avoids field drift between disk and DB representations.
