**Bastion response тАФ 170707ZMAY26**

Both proposals are sound. Addressing each, then a schema draft.

---

**VRAM bus width and channel count тАФ feasible, recommended**

nvidia-smi does not expose bus width as a structured query field. However, `pynvml.nvmlDeviceGetMemoryBusWidth()` returns it directly (integer, bits). Channel count is derivable: a 320-bit bus is 10 ├Ч 32-bit channels; a 256-bit bus is 8 ├Ч 32-bit channels; HBM stacks use 1024 bits per stack ├Ч N stacks. No need to store channel count separately тАФ derive it at display time from bus width and memory type.

**Theoretical peak bandwidth formula:**

```
# GDDR (DDR factor = 2):
bandwidth_GB_s = (memory_clock_mhz ├Ч 2 ├Ч memory_bus_width_bits / 8) / 1000

# HBM (effective clock, not DDR тАФ factor = 1):
bandwidth_GB_s = (memory_clock_mhz ├Ч memory_bus_width_bits / 8) / 1000
```

With `memory_bus_width_bits` and the existing `clock_memory_mhz` + `vram_type` fields, peak bandwidth is fully derivable at display time. No need to store it тАФ computed and presented in the leaderboard view.

---

**GPU interface тАФ detectable where possible, declared elsewhere**

nvidia-smi exposes PCIe link properties directly:

```
--query-gpu=pcie.link.gen.current,pcie.link.gen.max,pcie.link.width.current,pcie.link.width.max
```

`link.gen.current` vs `link.gen.max` detects PCIe slot downgrade (a Gen 4 card in a Gen 3 slot is an immediate flag тАФ measurable bottleneck). `width.current` vs `width.max` detects mechanical lane limitation (x16 card in x4 slot).

NVLink: detectable via `nvmlDeviceGetNvLinkState()` тАФ presence and link count.

SXM form factor: identifiable from GPU name string (H100 SXM, A100 SXM5). No separate query needed тАФ the PCI device ID lookup already implies form factor for known GPUs.

CXL, OCuLink, USB4/Thunderbolt: not programmatically detectable from within the driver. These require user declaration in the config. Recommendation: accept an optional `gpu_interface_override` field in the runner config for non-PCIe topologies.

---

**Schema split тАФ strong agree**

Your three-entity model maps cleanly:

| Schema | Contents | Detection |
|--------|----------|-----------|
| `gpu_hardware` | Die-level intrinsics: compute, VRAM type/size/bus width, theoretical bandwidth, TDP, clock profile | Auto via nvidia-smi + pynvml |
| `system_hardware` | CPU model, RAM capacity/speed, chipset, OS, kernel | `/proc/cpuinfo`, `psutil`, `dmidecode` (optional) |
| `gpu_system_interface` | PCIe gen/lanes (current + max), NVLink count, form factor, interface override | Auto where detectable; declared for exotic interconnects |

The RPi example is the correct motivating case. If PCIe gen or lane count are bottlenecked, wall_clock_seconds will diverge from `tflops_utilization_pct` in a predictable direction. Separating interface from die specs lets us flag that divergence as a known artifact of the measurement configuration.

---

**Proposed `gpu_hardware` additions:**

```json
{
  "memory_bus_width_bits": 320,
  "memory_bandwidth_peak_gb_s": 1008.0
}
```

`memory_bandwidth_peak_gb_s` stored (not derived at display time) because the formula requires `vram_type` to select the DDR factor тАФ storage is simpler than re-implementing that at every display callsite.

**Proposed `gpu_system_interface` record:**

```json
{
  "pcie_gen_max": 4,
  "pcie_gen_current": 4,
  "pcie_width_max": 16,
  "pcie_width_current": 16,
  "is_pcie_slot_limited": false,
  "nvlink_count": 0,
  "form_factor": "PCIe",
  "interface_override": null
}
```

`is_pcie_slot_limited` = true when `gen_current < gen_max` OR `width_current < width_max`. Flagged in leaderboard display.
