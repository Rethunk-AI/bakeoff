**Updated conflict detection policy — primary UUID already exists**

When `UUID5(BAKEOFF_MODEL_NAMESPACE, model_hash)` matches an existing DB row, run a field-by-field diff on the incoming disk file against the stored record:

**Soft fields — accept update silently:**
- `description`
- `creator.display_name`, `creator.homepage`, `creator.service_identifiers` (contact info evolution)

These carry no integrity constraint against the weights. An admin correction or upstream metadata refresh updating these is always valid.

**Hard fields — validate against weights before accepting:**
- `parameter_count_b`, `active_parameter_count_b`
- `context_length_default`, `context_length_min`, `context_length_max`
- `architecture`, `file_format`, `quantization`

If an incoming hard field differs from the stored value:
1. Re-derive the expected value from the weights file (same computation run at original ingest)
2. If incoming value matches derived value but stored value differs → the stored value was incorrect; accept the correction, update DB and disk
3. If incoming value does not match derived value → the source is invalid; reject the source, emit a clear error message indicating which field failed and what the derived value is
4. **Do not delete the disk file in any rejection case.** The submitter may have a typo or an in-progress edit; losing the file destroys their work. They retain the file, correct, and resubmit

**New version scenario:**

If multiple hard fields differ simultaneously and the incoming `version` field differs from the stored one, this is likely a distinct model variant rather than a correction. The ingest pipeline should surface a warning: "Incoming data differs from stored record on N fields. If this is a new version, create a new disk file with a distinct UUID rather than updating the existing entry." The source is not added; no file is deleted; the operator decides.

---

**Summary of conflict handling matrix**

| Scenario | Action |
|----------|--------|
| Hash match, soft fields differ | Accept update |
| Hash match, hard field matches weights but DB differs | Accept correction |
| Hash match, hard field does not match weights | Reject source, clear error, keep file |
| Hash match, multiple hard fields differ + version changed | Surface warning, block source, keep file |
| Hash match, new source URL | Add `model_sources` row only, no model update |

This fully covers the scenarios you described. Conflict detection policy is now settled. Will codify in the ingest pipeline spec when that work is scoped.
