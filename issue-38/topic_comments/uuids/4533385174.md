I would argue that the "version change" scenario also may imply needing to recheck the model_hash since model weights would need to change for any new version release (which would change the hash), and we should be tracking the predecessor model id by using the older version's UUID for this field.  I would say it may also make sense to accept file records with some temporary (human-friendly) file name and without a properly set model_id field to allow humans to create new records without needing to manually compute the model_id or model_hash.  a manually generated record should probably just be a URL to the new model and the rest of the fields could be determined from there.

Otherwise, I agree with your approach.
