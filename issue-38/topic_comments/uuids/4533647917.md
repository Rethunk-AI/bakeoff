**Q2 ‚Äî Hardware snapshot identity: A + C combined**

Confirmed reading. The two options are not mutually exclusive:

- Hardware specs are embedded inline in the results disk file (option C basis ‚Äî portable, air-gap compatible, no round-trip required)
- `hardware_id` UUID is computed deterministically from those embedded specs at ingest time (option A UUID generation) ‚Äî never stored in the disk file, derived on demand

This means disk files are fully self-contained; the UUID is an ingestion artifact, not a field the runner needs to compute or store.

**Proposed field list for `hardware_id` UUID input:**

System fields:
- `cpu_model` ‚Äî CPU make/model string
- `cpu_cores` ‚Äî physical core count
- `cpu_threads` ‚Äî logical thread count
- `ram_gb` ‚Äî total RAM in GB
- `motherboard` ‚Äî motherboard make/model

GPU fields (sorted list of tuples, one per slot):
- `gpu_model` ‚Äî GPU make/model string
- `vram_mb` ‚Äî VRAM in MB
- `slot_index` ‚Äî PCIe slot position

**BIOS / performance-affecting settings** ‚Äî you raise a valid point. These affect benchmark outcomes and should be part of the UUID if they materially differentiate two otherwise-identical machines. Proposed additional fields (all optional, null if not set or auto):

| Field | Description |
|-------|-------------|
| `memory_speed_mhz` | Active memory clock rate |
| `memory_channels` | Active memory channel count (1 = single, 2 = dual, 4 = quad) |
| `memory_interleave_profile` | XMP / EXPO / DOCP profile name or `manual` |
| `bios_notes` | Free-form for other performance-affecting settings (e.g., fTPM off, SMT off) |

`memory_speed_mhz`, `memory_channels`, `memory_interleave_profile` ‚Äî include in UUID input. `bios_notes` ‚Äî stored in snapshot but excluded from UUID (free-form field not stable for dedup). Does this field list look right, or are there other BIOS settings you want to capture or include in UUID generation?

**Q3 ‚Äî GPU/system snapshot disk layout: clarifying Option A, then recommendation**

Option A as proposed: a single `hardware-snapshot/<uuid>.json` that merges system fields and GPU list into one document ‚Äî no separate per-GPU files at the disk level. Example:

```json
{
  "schema_version": 1,
  "hardware_id": "<uuid5>",
  "cpu_model": "AMD Ryzen 9 7950X",
  "cpu_cores": 16,
  "cpu_threads": 32,
  "ram_gb": 64.0,
  "motherboard": "ASUS ROG Crosshair X670E",
  "gpus": [
    {"slot_index": 0, "gpu_model": "AMD Radeon RX 7900 XTX", "vram_mb": 24576, "interface": "PCIe 4.0 x16"}
  ]
}
```

Option B would split that into `gpu_hardware/<gpu_uuid>.json` and a separate system snapshot referencing GPU UUIDs.

**Recommendation given Q4 (run file embeds full hardware block):** Option A is the right shape for the `hardware-snapshot/` dedup cache ‚Äî it mirrors exactly what gets embedded in the run file. At ingest, the pipeline:
1. Extracts the hardware block from the run file
2. Computes `hardware_id` and per-GPU UUIDs
3. Writes `hardware-snapshot/<hardware_id>.json` if not already present (dedup cache)
4. Upserts DB rows for `system_hardware`, `gpu_hardware`, `system_gpu_link`

Separate `gpu_hardware/<gpu_uuid>.json` files (Option B) add a level of indirection that only pays off when many snapshots share the same GPU and you want to avoid re-embedding the spec. That optimization can land as the dedup enhancement in Phase 2 alongside the sparse run format.

**Q5 ‚Äî Runner hardware snapshot timing: CONFIRMED with caveat**

Option A confirmed ‚Äî snapshot once at process start, reuse across runs in same session. Consumer hardware does not hot-swap GPUs mid-session.

On interfaces: Thunderbolt 3/4, USB4, and eGPU enclosures are theoretically hot-swap capable (OS can detach/reattach the device). In practice, running a GPU benchmark mid-swap is not a real scenario. NVLink/SXM topologies cannot hot-swap. PCIe cannot hot-swap without platform-specific BIOS support.

Recommendation: Option A for Phase 1. If an operator is running on an eGPU enclosure and wants per-run snapshotting, they can restart the runner process. No need for a per-run snapshot path in the initial implementation.
