RE: bios_notes - power_limit_w: approved for metadata only; not in UUID.

RE: Go templates + sprig custom function plan: Schema DDL helpers: what if a column is removed and then added again, how would it detect that condition to allow for idempotency?

Agreed with Phase 1 scope: we only need some stubs in phase 1; we can implement the logic in phase 2, since that would be where we need the ability to migrate DB information.

RE: record_migration_script vs schema_migration_script: would the record_migration_script run for each record after the schema migration completes?  What if part of the record_migration_script involves changing the UUID field for that table?  I'm also in support of the 2 separate fields approach, but I want to verify this makes the most sense before confirming that design choice.

I think it makes sense for complex migrations with DDL updates to simply write all the data records back to their backing files (or a temp table) using the older format, do the DDL updates on an empty table, and then re-import the old format records through the record_migration_script into the new table. Simple migrations with DDL updates can just apply directly; This would be things like table/column renames, simple data conversions (int to stringified-int, int to float, timestamp to int, removing a NOT-NULL constraint, etc; things where there is no complex value changes or erroneous values to handle).  There can be a few helper functions to facilitate each of these flows.

I just had an idea - what if every UUID in the entire schema included some fixed-position indication of the table ID and possibly the schema version it came from as part of the UUID itself?  That would make UUIDs self-identifying if that information could be extracted from the UUID itself without needing to explicitly link it to a table through another field.  Is that a feasible design choice?  That would make migration much simpler as then we don't need UUIDs in the migrations table at all, and we can probably simplify the overall design.  Somehow I also thought the table_id + schema_version were identified in another table record somewhere as a unified ID of some kind, but maybe not?  Near the end of this discussion, let's go through all the tables schemas related to migration so I can see them together.

RE: uuid_migrations - version-based matching: I'm open to that approach, but there are concerns:
- How would we later update foreign keys in records from other tables that point to old UUID values after the FK target record has changed its UUID?  I believe that was the original goal behind this UUID_migrations table.
- How would we even recognize that the UUID an older record was pointing to in a FK is, in fact, an older UUID record and not a current version UUID?

If we can address these issues, I believe adding schema_version_id to each record in the disk file makes sense.  I think it also makes sense to assume the DB records at rest in the table are at the current version rather than explicitly storing that value in each record.  During a migration, the process can track based on the record's location (old table vs new table) what version it is and pass the appropriate value as if it were present as needed.  We do need to determine how to handle ingestion of initial user-created records in this case though; I propose assuming it applies to the current table version unless there is some obvious reason to assume otherwise.
