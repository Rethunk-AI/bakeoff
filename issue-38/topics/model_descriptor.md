**Bastion response — 201926ZMAY26**

@gissf1 — trustworthiness assessment and design implications.

---

**Ollama vs HuggingFace — trustworthiness**

Ollama is an acceptable source, roughly comparable to HuggingFace in trustworthiness. Key distinctions:

| Attribute | HuggingFace | Ollama |
|---|---|---|
| Source type | Direct from creator repos | Redistribution layer (mostly HF-sourced) |
| Model origin | Creator uploads GGUF, SafeTensors, etc. | Ollama team converts/mirrors from HF |
| Curation | Minimal — creator-controlled | Curated Ollama Library + community models |
| Commit tracking | HF commit hash per file | Ollama model tag (e.g., `llama3:8b-q4_K_M`) |
| Verification | SHA256 of weights file | Same SHA256 verification available |

Ollama models are largely the same underlying weights as HuggingFace — just repackaged into GGUF with specific quantization presets. For models in the Ollama Library (curated), provenance is traceable back to HF or the original creator. Community models have less formal provenance tracking.

---

**Design implications for our schema**

Our architecture already handles this cleanly. `model_sources` is provider-agnostic; `model_hash` (SHA256 of weights) is the deduplication ground truth regardless of where the model was obtained. An Ollama-sourced model with the same `model_hash` as a HF-sourced model will be detected as a duplicate and merged per the established workflow.

Recommendation: add `ollama` as a `source_types` entry alongside `huggingface`, `direct_url`, and `local_file`. No schema changes required — just a new row in the lookup table.

`source_metadata` JSONB for Ollama entries would store `{"ollama_tag": "llama3:8b-q4_K_M", "ollama_digest": "..."}`. Ollama exposes a digest per model pull that can serve as a secondary hash check before running the full `model_hash` computation.

---

**Caveat on community/unverified Ollama models**

Models outside the Ollama Library curated set have no guaranteed provenance chain. `model_hash` remains the authoritative trust anchor — source URL and tag are metadata, not trust signals. If the system eventually tracks a "trust tier" for sources, Ollama Library models and HF creator-published models would sit at the same tier; unverified community models from either platform would sit lower.

— Bastion

## 1. models table — existing columns, calculated VRAM

Confirmed: `parameter_count_b`, `active_parameter_count_b`, and `quantization` already exist. No new columns needed for those.

On `min_vram_gb` — agreed, do not store it. Calculate from existing fields:

```sql
-- Inline VRAM estimate for claim query
CEIL(
    m.active_parameter_count_b
    * CASE m.quantization
        WHEN 'fp32'    THEN 4.0
        WHEN 'fp16'    THEN 2.0
        WHEN 'bf16'    THEN 2.0
        WHEN 'int8'    THEN 1.0
        WHEN 'q8_0'    THEN 1.0
        WHEN 'q4_k_m'  THEN 0.45
        WHEN 'q4_0'    THEN 0.5
        WHEN 'q4_1'    THEN 0.5
        ELSE                 2.0   -- conservative default for unknown quant
      END
    * 1.15    -- 15% overhead for KV cache + activations
) AS estimated_vram_gb
```

Runner passes its available VRAM as a parameter; claim query filters `estimated_vram_gb <= $runner_vram_gb`. Calculation is per-row at claim time — no stored column, no staleness risk.

If quantization formats diverge from this list over time, the CASE expression is the single update point.

---

## 2. runners table — revised

Dropping hardware storage from runners. Runner declares its own capability at claim time. Table tracks identity, state, and heartbeat only:

```sql
CREATE TABLE runners (
    runner_id      TEXT        PRIMARY KEY,   -- stable agent ID (e.g. "worker-01")
    hostname       TEXT        NOT NULL,
    process_id     INT         NOT NULL,
    effective_user TEXT        NOT NULL,
    last_heartbeat TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    status         TEXT        NOT NULL DEFAULT 'ACTIVE'
                               CHECK (status IN ('ACTIVE','IDLE','DEAD')),
    started_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Runner registers on startup (INSERT or UPSERT). Heartbeats every 60s. Reaper marks DEAD when `last_heartbeat < NOW() - INTERVAL '5 minutes'` and reclaims their CLAIMED items.

`runner_id` in `runs` table: agreed — add `runner_id TEXT REFERENCES runners(runner_id)` to `runs`. Records which runner executed each run. Useful for diagnosing runner-specific failure patterns.

---

## 3. Claim query — revised

```sql
UPDATE run_queue
SET status = 'CLAIMED', claimed_by = $runner_id, claimed_at = NOW()
WHERE queue_id = (
    SELECT rq.queue_id
    FROM   run_queue rq
    JOIN   runs   r ON r.run_id   = rq.run_id
    JOIN   models m ON m.model_id = r.model_id
    WHERE  rq.status = 'PENDING'
      AND  (rq.retry_after IS NULL OR rq.retry_after <= NOW())
      AND  CEIL(
               m.active_parameter_count_b
               * CASE m.quantization
                   WHEN 'fp32'   THEN 4.0
                   WHEN 'fp16'   THEN 2.0
                   WHEN 'bf16'   THEN 2.0
                   WHEN 'int8'   THEN 1.0
                   WHEN 'q8_0'   THEN 1.0
                   WHEN 'q4_k_m' THEN 0.45
                   WHEN 'q4_0'   THEN 0.5
                   WHEN 'q4_1'   THEN 0.5
                   ELSE               2.0
                 END * 1.15
           ) <= $runner_vram_gb
    ORDER BY rq.priority ASC, rq.created_at ASC
    LIMIT 1
    FOR UPDATE SKIP LOCKED
)
RETURNING *;
```

`retry_after` replaces the `updated_at` gate entirely. Simpler: one boolean check per row vs. timestamp arithmetic. Agreed.

---

## 4. retry_after field — adopted

Accepting `retry_after TIMESTAMPTZ` over the `updated_at` math approach. Advantages you identified are correct:

- Simpler claim query (`retry_after <= NOW()` vs. `updated_at` delta math)
- Supports scheduling for non-failure reasons (marketing, legal, staged rollout)
- FAILED items stay FAILED — high `attempt_count` with no PENDING transition = visible signal for operator investigation

**Revised retry flow:**
1. Run → FAILED. Set `retry_after = NOW() + base_interval`, bump `attempt_count`, bump `priority`.
2. Item stays FAILED. Claim query ignores it (status ≠ 'PENDING').
3. Reaper (probabilistic, runner-side) finds: `status = 'FAILED' AND attempt_count < max_attempts AND retry_after <= NOW()` → reset to PENDING, clear `retry_after`.
4. Item re-enters claim pool.

`base_interval` for retry: 5 minutes is fine as a floor. Reaper/runner can use exponential: `5 * 2^(attempt_count - 1)` minutes, capped at e.g. 60 minutes.

Recommendation: stay TEXT + CHECK now. Convert to ENUM when the status set is stable and Phase 1 schema work closes. Mark as a deferred cleanup item.

The DB manages its own row-level timestamps via trigger; the disk file carries its own copy so the file is self-contained. If the model's publication date is relevant, that is `release_date` in both DB and disk, sourced from upstream.

**FK auto-add policy — confirmed**

Confirmed. When an ingested model disk file contains a text value (architecture, file_format, quantization) with no matching lookup row:

1. Reject the ingest
2. Log an admin alert with the unrecognized value
3. Hold the submission in `pending_review` state
4. Admin reviews, adds the lookup entry manually, re-runs ingest

No automated addition to lookup tables. Unreviewed additions would corrupt filter values, VRAM estimates (quantization case), and UI filter bar populations.

---

**model_id UUID — conflict detection + multi-source coalescing**

When weights are pulled and `model_hash` is computed:

1. Generate primary UUID: `UUID5(BAKEOFF_MODEL_NAMESPACE, model_hash)`
2. Check DB for existing row with either UUID (provisional or primary)
3. Provisional exists → update `model_id` to primary UUID, mark non-provisional, rename disk file
4. Primary already exists (same hash, new source) → do not create new `models` row; add new `model_sources` entry instead

Multiple provisionals for same model from different sources: when the first provisional's weights are pulled and primary UUID is generated, the DB now has the primary entry. When the second provisional's weights are pulled: hash match → detected as duplicate model → source added to `model_sources`, provisional disk file removed. Coalescing is automatic at hash-computation time.

Provisional added after primary exists: ingest pipeline checks for existing primary before creating provisional. Source URL already in `model_sources` → reject (duplicate). Source URL new but hash will match → create provisional marked `pending_hash_resolution`, coalesce at weight-pull.

When `UUID5(BAKEOFF_MODEL_NAMESPACE, model_hash)` matches an existing DB row, run a field-by-field diff on the incoming disk file against the stored record:

**Soft fields — accept update silently:**
- `description`
- `creator.display_name`, `creator.homepage`, `creator.service_identifiers` (contact info evolution)

These carry no integrity constraint against the weights. An admin correction or upstream metadata refresh updating these is always valid.

**Hard fields — validate against weights before accepting:**
- `parameter_count_b`, `active_parameter_count_b`
- `context_length_default`, `context_length_min`, `context_length_max`
- `architecture`, `file_format`, `quantization`

If an incoming hard field differs from the stored value:
1. Re-derive the expected value from the weights file (same computation run at original ingest)
2. If incoming value matches derived value but stored value differs → the stored value was incorrect; accept the correction, update DB and disk
3. If incoming value does not match derived value → the source is invalid; reject the source, emit a clear error message indicating which field failed and what the derived value is
4. **Do not delete the disk file in any rejection case.** The submitter may have a typo or an in-progress edit; losing the file destroys their work. They retain the file, correct, and resubmit

**New version scenario:**

If multiple hard fields differ simultaneously and the incoming `version` field differs from the stored one, this is likely a distinct model variant rather than a correction. The ingest pipeline should surface a warning: "Incoming data differs from stored record on N fields. If this is a new version, create a new disk file with a distinct UUID rather than updating the existing entry." The source is not added; no file is deleted; the operator decides.
