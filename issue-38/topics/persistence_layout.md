**Bastion response — 200327ZMAY26**

@gissf1 — addressing all points in order.

---

**A1: Prompt FK to task_id + task/test/test suite terminology**

Agreed — a prompt record should have a `task_id` FK. This also lets `run_model_metrics` drop the separate `task_id` field: prompt implies task.

Terminology disambiguation:

| Term | Definition |
|---|---|
| **Test suite** | A named collection of tasks run together (e.g., "baseline tier", "comparison tier"). Maps to a top-level task record with `parent_id IS NULL`. |
| **Task** | A specific evaluation type within a suite (e.g., "bash_scripting", "code_generation_python"). Corresponds to a single evaluation scenario, potentially with multiple prompt variants. |
| **Prompt** | A specific text formulation of a task. One task can have many prompt variants (for A/B testing, difficulty levels, etc.). The prompt is what gets sent to the model. |
| **Test run** | One execution of a prompt against one model on specific hardware. Produces `run_model_metrics` + `run_hardware_metrics` rows. |

"Test" as a term is overloaded — I'll avoid it in schema names and use "task", "prompt", and "run" as the canonical vocabulary.

---

**A2: File-based prompts + DB metadata — agreed**

File-based prompt content is correct: version control is the right persistence layer for prompt text. Git gives us full history, diff, and blame for free.

DB metadata stored per prompt:
- `file_path` — relative path within the repo
- `git_commit_hash` — SHA of the commit in which this prompt was last modified (deterministic unique identifier for this prompt version)
- `content_length_bytes` — byte count of the raw prompt text
- `version` — human-readable version label (e.g., "1.0", "1.1-prerelease")
- `release_date` — date this version was officially released
- `is_prerelease` — flag for prompts in testing before official release
- `task_id` FK — which task this prompt belongs to

For reproduction: given a `run_model_metrics` row, the `git_commit_hash` on its referenced `prompt` record uniquely identifies the exact prompt text used. Historical versions accessible via git. No need to store prompt text in the DB.

---

**A3: Model deduplication + versioning**

Source URL as primary identifier: reasonable starting point, but insufficient alone — HuggingFace and Ollama can host the same weights under different URLs.

Proposed `models` table:

```
models
──────────────────────────────────────────────────────────
model_id                SERIAL PRIMARY KEY
name                    TEXT NOT NULL              -- user-friendly name: "Phi-3.5-mini-instruct"
creator                 TEXT NULLABLE              -- "microsoft", "meta", "mistralai"
source_urls             TEXT[] NOT NULL            -- all known source URLs (HF, Ollama, etc.)
model_hash              TEXT NULLABLE              -- SHA256 of model weights file; deduplication ground truth
release_date            DATE NULLABLE
version                 TEXT NULLABLE              -- model version label if versioned
description             TEXT NULLABLE              -- scraped or manually entered
predecessor_model_id    INT NULLABLE REFERENCES models  -- lineage tracking

-- Architecture / capability
parameter_count         BIGINT NULLABLE            -- total params
active_parameter_count  BIGINT NULLABLE            -- active params (MoE only)
architecture            TEXT NULLABLE              -- "Dense", "MoE"
context_length_default  INT NULLABLE
context_length_min      INT NULLABLE
context_length_max      INT NULLABLE
file_format             TEXT NULLABLE              -- "GGUF", "SafeTensors", etc.
quantization            TEXT NULLABLE              -- "Q4_K_M", "fp16", etc.
min_vram_mb             INT NULLABLE               -- minimum VRAM required at runtime
```

"provider" → "creator" (or "developer"). These are locally-hosted models; the model's creator is the research lab or team, not a cloud provider.

Deduplication strategy (in preference order):
1. `model_hash` match — authoritative (same weights = same model regardless of source)
2. HuggingFace model ID match — reliable for HF-sourced models; parse from source URL
3. Name + creator + quantization match — fallback heuristic; flag for human review

URL versioning: if a HF model URL resolves to different weights at different times, the `git_commit_hash` on the HF model repo differentiates versions. Recommend storing a `hf_commit_hash` field alongside `model_hash` for HF-sourced models.

---

**Q4: Score + pass/fail both required; hardware-neutral**

Agreed on both points:
- Both `score` (float 0–1) and `pass_fail` (boolean) stored per run
- Score and pass/fail are model-capability metrics only — hardware limitations (OOM, timeout) are a `failure_reason`, not a scoring event. A model that times out gets `failure_reason = "timeout"`, not a failing score.
- The combined hardware+quality efficiency metric is a Phase 2 derived view, not a stored field. It can be computed from `run_model_metrics × run_hardware_metrics` at display time.

---

**Q5: Runs table + natural key vs surrogate + prompt git hash**

A `runs` table is needed. Fields that are run-scoped (timestamp, submitter, runner commit hash) don't belong on `run_model_metrics` or `run_hardware_metrics` — those tables describe what was measured, not who ran it or when.

Proposed `runs` table:

```
runs
──────────────────────────────────────────────────────────
run_id                  UUID PRIMARY KEY           -- generated at run time; natural key
submitted_at            TIMESTAMPTZ NOT NULL
publisher_id            TEXT NOT NULL              -- submitting user/account
runner_version          TEXT NULLABLE              -- bakeoff harness semver or commit hash
prompt_git_hash         TEXT NULLABLE              -- git commit hash of prompt files at run time
```

UUID as `run_id` is the right choice over serial integer: UUIDs are globally unique without a database sequence, can be generated client-side before the DB write, and prevent enumeration of run IDs by outsiders. Natural key (the UUID itself) is sufficient; no surrogate needed.

`prompt_git_hash` here is the harness-level hash (which commit of the prompts repo was in use at run time), distinct from the per-prompt `git_commit_hash` on the `prompts` table. Both are useful: the prompts table hash pinpoints which version of a specific prompt was used; the run-level hash pins the entire prompt set version.

`run_model_metrics` and `run_hardware_metrics` both add a `run_id FK → runs`.

---

**Preliminary table comments**

`models.creator` over `provider` — confirmed above. "developer" also acceptable; "creator" is one word shorter and unambiguous.

**Tiered task structure** — agreed. Proposed:

```
task_categories
──────────────────────────────────────────────────────────
category_id             SERIAL PRIMARY KEY
name                    TEXT NOT NULL UNIQUE       -- "baseline", "comparison", "advanced"
description             TEXT NULLABLE

tasks
──────────────────────────────────────────────────────────
task_id                 SERIAL PRIMARY KEY
name                    TEXT NOT NULL
category_id             INT NULLABLE REFERENCES task_categories
parent_id               INT NULLABLE REFERENCES tasks  -- null = top-level (suite)
sort_order              INT NOT NULL DEFAULT 0     -- execution order within parent
description             TEXT NULLABLE
```

Top-level tasks (`parent_id IS NULL`) are suites. Sub-tasks are the individual evaluation types. `sort_order` controls execution sequence.

**File-based tasks/prompts vs DB-centric** — hybrid is correct:
- Task *definitions* (name, category, sort order, parent) live in the DB. These change rarely and are queryable for scheduling and reporting.
- Prompt *content* lives in files (git-tracked). DB stores metadata pointing to the file version.
- No redundancy: task rows don't contain prompt text; prompt rows don't embed task logic.

**`run_model_metrics.task_id`** — drop it if `prompt_id` FK already implies `task_id` via `prompts.task_id`. The task is derivable: `run → prompt → task`. Denormalization is only warranted if that join is a hot query path (e.g., "all runs for task X regardless of prompt variant"). Recommendation: omit for now; add a `task_id` column to `run_model_metrics` only if query profiling shows the join is costly.

**Testing queue** — agreed, defer to a new issue. Standing by to open it on direction.

---

**Task/Prompt cardinality — one task, many prompts**

One task can have many prompts. Your instinct is correct. Industry standard:

- **HumanEval:** 164 individual code problems. Each problem is one prompt. "Code generation" is the task category; each problem is a prompt instance of that task.
- **IFEval:** 541 prompts testing instruction-following patterns. Prompts are organized by instruction type (e.g., "include keyword X", "respond in under N words"). The instruction type = task; each test instance = one prompt.
- **MMLU-Pro:** Questions organized by subject area (medicine, law, etc.). Subject area = task category; each question = one prompt.

## Proposed: `run_queue` table

```sql
CREATE TABLE run_queue (
    queue_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    run_id          UUID NOT NULL REFERENCES runs(run_id) ON DELETE CASCADE,
    model_id        INT  NOT NULL REFERENCES models(model_id),
    prompt_set_id   INT  NOT NULL,              -- logical grouping; FK TBD when prompts table extended
    hardware_tag    TEXT NOT NULL,              -- "gpu-a100", "cpu-only", etc. — matches runner affinity
    priority        INT  NOT NULL DEFAULT 100,  -- lower = higher priority; 0 = critical
    status          TEXT NOT NULL DEFAULT 'PENDING'
                        CHECK (status IN ('PENDING','CLAIMED','IN_PROGRESS','COMPLETE','FAILED','CANCELLED')),
    attempt_count   INT  NOT NULL DEFAULT 0,
    max_attempts    INT  NOT NULL DEFAULT 3,
    claimed_by      TEXT,                       -- runner identity (hostname + PID)
    claimed_at      TIMESTAMPTZ,
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    error_detail    TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index for next-item-to-claim query
CREATE INDEX run_queue_claim_idx
    ON run_queue (hardware_tag, priority, created_at)
    WHERE status = 'PENDING';
```

---

## State machine

```
PENDING → CLAIMED → IN_PROGRESS → COMPLETE
                  ↘             → FAILED  → PENDING (if attempt_count < max_attempts)
                                           → FAILED  (terminal, if attempt_count >= max_attempts)
PENDING → CANCELLED (operator cancel)
CLAIMED → PENDING  (claim timeout: >10 min in CLAIMED without IN_PROGRESS transition = stale claim)
```

**Claim protocol (atomic, race-safe):**
```sql
UPDATE run_queue
SET status = 'CLAIMED', claimed_by = $runner_id, claimed_at = NOW()
WHERE queue_id = (
    SELECT queue_id FROM run_queue
    WHERE status = 'PENDING'
      AND hardware_tag = $runner_hardware_tag
    ORDER BY priority ASC, created_at ASC
    LIMIT 1
    FOR UPDATE SKIP LOCKED
)
RETURNING *;
```

`FOR UPDATE SKIP LOCKED` is the standard Postgres queue-claim idiom. Multiple runners on the same hardware tag cannot claim the same item.

---

## Priority ordering

| Priority value | Meaning |
|---------------|---------|
| 0–9 | Critical (baseline tier, blocks comparisons) |
| 10–49 | High (new model, first submission) |
| 50–99 | Normal (scheduled refresh) |
| 100+ | Low (re-run for additional samples) |

**Dependency ordering:** baseline tier items enqueued at priority 0–9. Comparison-tier items enqueued at 50. Runner responsible for checking whether baseline is COMPLETE before marking comparison IN_PROGRESS (or queue can enforce via FK / trigger — open question for @gissf1).

---

## Retry policy

- `max_attempts = 3` default.
- Retry trigger: on FAILED with `attempt_count < max_attempts`, reset to PENDING, increment `attempt_count`, set `priority = priority + 10` (deprioritize relative to fresh items).
- No backoff delay in DB — runner responsible for exponential backoff before re-claiming.
- OOM / timeout failures: same retry path; `error_detail` records cause.
- Terminal failure (`attempt_count >= max_attempts`): stays FAILED. Manual operator action to reset or cancel.

---

## Persistence: DB-only (proposed)

Git-tracked file queue rejected. Rationale:
- `FOR UPDATE SKIP LOCKED` requires Postgres; file-based queues cannot provide race-safe multi-runner claims.
- Queue state is transient operational data, not result data. Git is append-only history; queue state is mutable — wrong abstraction.
- Schema.sql is already the substrate. Adding `run_queue` keeps all operational state in one plane.

If git-tracked queue is preferred for audit/portability reasons, @gissf1 to advise — Bastion will revise.

---

## Runner interaction

```
1. Runner starts → declares hardware_tag + runner_id
2. Claims next PENDING item (atomic UPDATE above)
3. Transitions to IN_PROGRESS (UPDATE status='IN_PROGRESS', started_at=NOW())
4. Executes run → writes results to run_model_metrics
5. Transitions to COMPLETE (UPDATE status='COMPLETE', completed_at=NOW())
   OR FAILED (UPDATE status='FAILED', error_detail=<cause>)
6. Retry logic: if FAILED and attempt_count < max_attempts → reset to PENDING
7. Loop back to step 2
```

Heartbeat: runner should UPDATE `updated_at` every 60s while IN_PROGRESS. Stale-claim reaper (cron or background goroutine) resets items where `status='CLAIMED'` and `claimed_at < NOW() - INTERVAL '10 minutes'`.

---

## Multi-runner support

Fully supported via `FOR UPDATE SKIP LOCKED`. Runners filter by `hardware_tag` — a GPU runner will not claim CPU-only items and vice versa. No coordinator process required.

## 7. Q4 — reaper: runner-side, probabilistic

Your 10% random reaper proposal is sound. Formalized:

**Runner loop:**
1. Claim next compatible PENDING item (claim query in §1).
2. Transition to IN_PROGRESS → execute → COMPLETE or FAILED.
3. After any outcome: `if rand() < 0.10 { run_reaper() }`.
4. Heartbeat: UPDATE runners SET last_heartbeat = NOW(), status = 'ACTIVE' every 60s.

These fields describe the lifecycle of the *disk record*, not the model weights or the upstream publication date.

| `add_column <table> <col> <type>` | No-op if column already exists |
| `drop_column <table> <col>` | No-op if column does not exist |
| `rename_column <table> <old> <new>` | No-op if `new` already exists and `old` is absent |
| `add_table <table> <sql>` | No-op if table already exists |
| `drop_table <table>` | No-op if table does not exist |

Scripts run in strict version order; the version tracking table ensures each script runs exactly once (except on crash recovery re-run, where the no-op rules make re-execution safe). The "remove X in vN, add X back in vM" case is entirely handled by sequencing: vN's `drop_column X` removes it, vM's `add_column X <new_type>` adds it back. No collision.

On `add_column` vs type change: if a column exists with the wrong type, `add_column` skips it silently. Type changes require an explicit `alter_column_type` helper (to be added to the function set). This makes the DDL intent unambiguous: `add_column` is append-only; `alter_column_type` is deliberate mutation.

Phase 1 scope confirmed: stub function signatures now. Implementation in Phase 2. `schema_versions` table present in Phase 1; first migration scripts appear when schema evolves.

---

**record_migration_script — per-record timing and UUID change**

**Timing:** `record_migration_script` runs at ingest time, per old-format record, lazily. When an old disk file is ingested (schema_version_id < current), the pipeline applies the script to the record before writing to DB. Schema migration (DDL) runs first, once, via `schema_migration_script`. Record migration runs after, per record, at ingest.

**For complex DDL migrations** where in-place lazy migration is infeasible (e.g., column type changes affecting all rows), your proposed pattern is correct and now canonical:

1. Write all affected records back to backing disk files using the old format
2. Apply DDL to the empty table (`schema_migration_script`)
3. Re-import all records through `record_migration_script` into the updated table

This is the batch path. Simple migrations (renames, type widening, removing NOT NULL, adding nullable columns) apply directly via `schema_migration_script` alone; no record re-export needed.

**If record_migration_script changes UUID-input fields:** the migration pipeline contract is:
1. Compute `old_uuid` from original disk file fields (before script runs)
2. Apply `record_migration_script`
3. Compute `new_uuid` from transformed fields (using current schema namespace — see below)
4. If `old_uuid ≠ new_uuid`: write entry to `uuid_migrations` automatically

This is enforced by the pipeline, not by the script author. Migration scripts declare field transformations; the pipeline handles UUID bookkeeping.

Two separate fields (`schema_migration_script` + `record_migration_script`) confirmed correct.

---

**Self-identifying UUIDs via UUID v5 namespaces — YES, feasible**

The idea is sound. Implementation: each schema version defines a fixed UUID v5 namespace derived from a version string:

```
NAMESPACE_V1 = uuid5(UUID_NIL, "bakeoff.hardware_uuid.v1")
NAMESPACE_V2 = uuid5(UUID_NIL, "bakeoff.hardware_uuid.v2")
```

Hardware fingerprint UUID for version N = `uuid5(NAMESPACE_VN, canonical_fields_json)`.

Two properties this gives:
1. **Version identification:** given an unknown UUID and a set of known namespaces, you can determine which schema version generated it by re-deriving it against each namespace. No separate version tag on the UUID is needed.
2. **Collision avoidance:** same hardware fingerprint in v1 and v2 produces different UUIDs, so there is no ambiguity.

In practice, the disk file's `schema_version_id` already tells you which namespace to use — you don't need to brute-force-test. But the namespace embedding means the UUID is still self-identifying if the disk file metadata is lost or unavailable.

**Does this eliminate uuid_migrations?** Partially. For disk-file-to-DB ingest, you can always recompute the new UUID from the disk file + current namespace — no uuid_migrations lookup needed. But uuid_migrations is still needed for **FK resolution across tables**: if table A holds a FK to table B using a v1 UUID, and B migrated to v2, the FK value in A is stale. uuid_migrations bridges old_uuid → new_uuid for those cross-table references. The table is not eliminated, but its role is narrowed to FK resolution rather than version identification.

Recommendation: adopt UUID v5 namespaces per your idea. uuid_migrations retains its FK bridge role only.

---

**uuid_migrations — FK concerns**

Both concerns addressed:

**1. How to update FKs in other tables pointing to old UUID values?**

Lazy resolution — no eager FK cascade. The pipeline resolves old UUIDs to new at ingest time:
- Encounter old UUID in FK field → look up `uuid_migrations` (old_uuid, entity_type) → use new_uuid
- For EXISTING DB records with stale FK values: a one-time batch migration pass at schema upgrade time iterates each FK column, resolves via uuid_migrations, and writes the current UUID. This runs once per upgrade as part of the schema_migration_script.

**2. How to recognize that a FK's UUID is an old version vs current?**

With UUID v5 namespaces: test the UUID against `NAMESPACE_VN` for each known version. If it matches an older namespace, it is an old UUID. If it matches the current namespace, it is current.

Without namespace testing: uuid_migrations lookup. If found in uuid_migrations as old_uuid → stale. If not found there AND found in the primary table → current.

```sql

CREATE TABLE schema_versions (
    schema_version_id       SERIAL PRIMARY KEY,
    description             TEXT NOT NULL,
    schema_migration_script TEXT,   -- DDL; runs once per upgrade; idempotent; NULL = no-op

);

-- Tracks which versions have been applied to this DB instance
CREATE TABLE schema_version_history (
    schema_version_id INTEGER NOT NULL REFERENCES schema_versions(schema_version_id),
    applied_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (schema_version_id)
);

-- Bridges old UUIDs to current UUIDs for FK resolution across tables
CREATE TABLE uuid_migrations (
    migration_id  SERIAL PRIMARY KEY,
    entity_type   TEXT NOT NULL,   -- e.g. 'system_hardware', 'gpu_hardware'
    old_uuid      UUID NOT NULL,
    new_uuid      UUID NOT NULL,
    from_version  INTEGER NOT NULL REFERENCES schema_versions(schema_version_id),

    UNIQUE (entity_type, old_uuid)
);

-- Canonical list of tables with UUID-based identity (for entity_type FK)

    table_id    SERIAL PRIMARY KEY,
    table_name  TEXT NOT NULL UNIQUE,
    uuid_namespace_v1 UUID NOT NULL  -- base namespace for UUID v5 generation
);
```

**Self-identifying UUID — namespace concern: VALID, design corrected**

The concern is correct. Per-version namespaces (NAMESPACE_V1, NAMESPACE_V2, ...) force a UUID recomputation for every record in a table on every schema version bump, even when that record's UUID-input fields did not change. At scale, this produces uuid_migrations growth proportional to (records × version count) — unbounded and unacceptable.

**Corrected design: per-table namespace, not per-version.**

`schema_tables.uuid_namespace` is a fixed UUID per table, set once at table creation and never changed. It identifies the owning table; it does not encode the schema version.

UUID v5 = `uuid5(TABLE_NAMESPACE, canonical_fields_json)` where `canonical_fields_json` is the sorted, deterministic serialization of the UUID-input fields at the schema version active when the record was created.

When a new schema version adds a UUID-input field:
- Records where the new field produces a different canonical string → new UUID (migration entry created)
- Records where the new field is absent or matches the assumed default → **same canonical string, same UUID** — no migration entry needed

Growth in uuid_migrations is now proportional to (affected records × UUID-input field changes), not to (all records × version count). Records with stable UUID-input fields are never touched.

---

**Eliminating UUIDs from uuid_migrations — approved**

Agreed. The per-record UUID storage approach has the unbounded-growth problem you identified. With per-table namespaces and schema_version_id embedded in each record, FK resolution becomes computable rather than looked-up.

**FK resolution without uuid_migrations UUIDs:**

When an FK UUID is encountered that does not match any current record:
1. Read the referencing record's `schema_version_id` — this bounds the age of the UUID
2. From FK field name + `schema_tables`, identify the target table and its namespace
3. Narrow candidate versions to the range [referencing_record_version, current_version]
4. Re-derive the UUID at each candidate version against the known namespace — the set is small (typically 1–2 hops), not a full brute-force
5. Match found → use that version's record; no match → flag for operator review

This avoids the uuid_migrations explosion entirely. The old_uuid/new_uuid columns are removed.
