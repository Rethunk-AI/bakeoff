### #8 comment 4467940739 — @AlbinoGeek — 2026-05-16T19:52:07Z
<https://github.com/Rethunk-AI/bakeoff/issues/8#issuecomment-4467940739>

**Bastion response — 161950ZMAY26**

All four additions are feasible. Addressing each:

---

**VRAM size and type**

`memory.total` is already collected (partially — it's in `GpuSample`). Exposing it to the `hardware_specs` record is a schema addition, not new collection work.

Memory type (GDDR6, GDDR6X, HBM2e, etc.) is not directly exposed by nvidia-smi in a structured field. Two approaches:
1. `pynvml.nvmlDeviceGetMemoryInfo()` — returns size only, not type.
2. Device ID lookup table: given `pci.device_id`, the GPU model is deterministic — GDDR6X on RTX 3080/4090, HBM2e on A100, GDDR6 on RTX 3060, etc. Maintain a lookup keyed by PCI device ID. This is the practical path.

---

**Clock rates (current, max, boost)**

nvidia-smi exposes these directly:

```
--query-gpu=clocks.current.memory,clocks.max.memory,clocks.current.graphics,clocks.max.graphics,clocks.boost.graphics
```

Overclocking detection: if `clocks.current.graphics > clocks.boost.graphics` at peak inference load, card is running above factory boost. Flag as `is_overclocked: true` with the delta recorded. Reliable signal for cards running third-party OC profiles (MSI Afterburner, etc.).

---

**PCI IDs**

nvidia-smi provides full identifying data:

```
--query-gpu=gpu_bus_id,pci.device_id,pci.sub_device_id,pci.bus_id
```

`pci.device_id` uniquely identifies the GPU die. `pci.sub_device_id` identifies the board partner variant (EVGA vs ASUS vs Founders Edition — same die, different thermal/power design). Recording both captures full hardware identity, not just the chip.

---

**Submitter identity**

`publish.py` already accepts a publisher identity from the submitting user. Recommend adding a `system_id` field alongside `publisher_id`:
- `publisher_id` — the user or account submitting the result (already tracked)
- `system_id` — a UUID generated once per test host at first run, stored in local config. Stable across runs on the same hardware. Allows grouping results by machine even if the same hardware is submitted by multiple accounts.

Combined, `publisher_id` + `system_id` + `pci.device_id` creates a three-factor identity that makes result fabrication detectable: a claimed `hardware_id` of `rtx-4090-24gb` from a `pci.device_id` that maps to a 3060 Ti is an immediate flag.

---

**Schema additions for `hardware_specs`:**

```json
{
  "hardware_id": "rtx-4090-24gb",
  "pci_device_id": "0x2684",
  "pci_sub_device_id": "0x3984",
  "pci_bus_id": "00000000:01:00.0",
  "vram_total_mb": 24576,
  "vram_type": "GDDR6X",
  "clock_memory_mhz": 10501,
  "clock_graphics_current_mhz": 2520,
  "clock_graphics_boost_mhz": 2520,
  "is_overclocked": false,
  "system_id": "uuid-stable-per-host"
}
```

Will open sub-issues for implementation if directed.

### #8 comment 4472501126 — @AlbinoGeek — 2026-05-17T21:04:56Z
<https://github.com/Rethunk-AI/bakeoff/issues/8#issuecomment-4472501126>

**Bastion response — 172104ZMAY26**

Addressing each point in order:

---

**Indexes on `system_gpu_link`**

Agreed. Since the PK is `(system_hardware_id, slot_index)` (see below), the composite PK index covers lookups by `system_hardware_id` already. `gpu_hardware_id` is data — no implicit index — and will be queried independently (e.g., "all systems running this GPU model"), so a standalone index on `gpu_hardware_id` is correct.

---

**PK revision: `(system_hardware_id, slot_index)`**

Correct. A slot is a fixed property of the system. At any point in time, a given slot holds at most one GPU — so the GPU identity is data about the slot, not part of the key. The PK from my prior schema `(system_hardware_id, gpu_hardware_id, slot_index)` was wrong: it would prevent the same GPU model from occupying two slots in the same system (two identical cards), which is a valid and common configuration.

Revised `system_gpu_link`:

```
system_gpu_link
───────────────────────────────────────────────────
system_hardware_id          FK → system_hardware
slot_index                  INT NOT NULL
PK: (system_hardware_id, slot_index)

gpu_hardware_id             FK → gpu_hardware      (indexed)
gpu_native_interface_type_id  FK → interface_type  -- GPU card's rated spec
slot_actual_interface_type_id FK → interface_type  -- what the slot is running
```

---

**`is_slot_limited` — drop it**

Derivable: `gpu_native_interface_type_id ≠ slot_actual_interface_type_id`. PostgreSQL `GENERATED ALWAYS AS` does not support subquery-based expressions, so a truly computed column is not viable here. Storing it requires a trigger. Neither is worth the complexity. Recommendation: drop the column entirely; compute the condition in queries and views. Cheap comparison on indexed FKs.

---

**Field naming: "native" ambiguity**

You are correct. "Native" is ambiguous — it could describe the system slot's native spec or the GPU card's native spec. Revised names applied above:

- `gpu_native_interface_type_id` — what the GPU card itself is rated for (GPU's spec)
- `slot_actual_interface_type_id` — what the slot is actually running at (observed state)

Clearer at a glance which side the constraint comes from.

---

**Degraded description — runtime concatenation vs stored rows**

The theoretical question is valid: does a "PCIe 3.0 x16 link in a PCIe 4.0 x16 slot" yield exactly PCIe 3.0 x16 bandwidth? In practice: yes, the slower side governs, and the bandwidth equals the lower-rated interface spec. Exceptions (signal integrity issues, driver overhead) would show up as reduced measured throughput in `run_hardware_metrics`, which is the right place for that data.

For display: recommend runtime concatenation rather than exhaustively pre-seeded degraded-state rows. The number of possible native/actual combinations is large; pre-seeding every pairing is impractical. The `description` field from each `interface_type` row is already human-readable. Presentation layer constructs: `"[slot_actual.description] (limited from [gpu_native.description])"` when native ≠ actual. `bandwidth_peak_gb_s` from `slot_actual_interface_type_id` is the authoritative number for filtering and comparison.

---

**`run_hardware_metrics` → `system_gpu_link`**

Agreed — reference the `(system_hardware_id, slot_index)` compound key (or a surrogate if added). Full hardware context for any run is then: `run_hardware_metrics → system_gpu_link → system_hardware + gpu_hardware + interface_type`.

---

**Multi-GPU runs — defer**

Agreed. When multi-GPU distribution becomes relevant, the natural extension is a `run_gpu_usage` join table: `(run_id, system_hardware_id, slot_index)` referencing `system_gpu_link`. That design is forward-compatible with the revised PK above. No changes needed now — log it as a Phase 2 schema extension.

---

**Multi-system runs — defer**

Agreed. The "composite system" concept for distributed runs across physical machines is valid but niche. No schema work in Phase 1.

---

**bakeoff-results documentation — hold**

Agreed. Will update once this design is finalized and ratified.

**`run_model_metrics` schema — models, tasks, prompts, and scoring**

Opened per discussion in #8 (hardware/performance metrics). This issue covers the model-side schema: how models, tasks, and prompts are represented and how `run_model_metrics` references them.

---

## Current target schema (placeholder — FKs not yet resolved)

```
run_model_metrics
──────────────────────────────────────────────────────────
run_id                          TEXT NOT NULL PRIMARY KEY
model_id                        TEXT NOT NULL      -- FK to models (TBD)
task_id                         TEXT NOT NULL      -- FK to tasks (TBD)
prompt_id                       TEXT NOT NULL      -- FK to prompts (TBD)
score                           FLOAT NULLABLE
pass_fail                       BOOLEAN NULLABLE
flops_per_token_theoretical     BIGINT NULLABLE
```

`model_id`, `task_id`, `prompt_id` are currently TEXT placeholders. This issue resolves them into proper entity tables with FK references.

---

## Open questions for discussion

**Q1 — Task / prompt relationship**

Working assumption: a *task* is an evaluation scenario type (e.g., "code generation", "multi-step reasoning", "summarization"). A *prompt* is a specific text formulation of that task. One task may have many prompt variants. Is this correct?

**Q2 — Prompt data storage: database or file?**

Two options:

- **In database:** `prompts` table stores prompt text. Submitted results reference `prompt_id` FK — leaderboard can verify submitted results against the exact input used. Enables full reproducibility audit.
- **In files:** prompts stored in versioned benchmark config files. `prompt_id` is a content hash or file path reference. Simpler runner integration; requires tracking file version alongside result.

Which path is preferred? This is load-bearing for leaderboard result verification.

**Q3 — Model identity**

What constitutes a distinct `model_id`? Options:
- Provider + model name + version string (e.g., `openai/gpt-4o-2024-11-20`)
- A structured `models` table with separate fields for provider, model_family, version, parameter_count, quantization, context_length, etc.

A structured `models` table enables hardware-normalized comparisons across quantization variants of the same base model. Is that level of detail needed in Phase 1?

**Q4 — Scoring schema**

`score FLOAT` and `pass_fail BOOLEAN` are both present. Are these always computed from the same test, or do some tasks produce only a score (no binary), and others only pass/fail (no continuous score)? Should these be separated into different record types, or is nullable acceptable?

**Q5 — Link to run_hardware_metrics**

`run_hardware_metrics` and `run_model_metrics` both reference `run_id`. Is `run_id` a natural key (e.g., UUID generated per invocation), or should there be an explicit `runs` table that both tables FK into? A `runs` table would carry: timestamp, submitter, runner_version, and serve as the join anchor for all per-run data.

---

## Proposed starting point (pending Q1–Q5 answers)

### #12 comment 4494171918 — @AlbinoGeek — 2026-05-20T03:17:33Z
<https://github.com/Rethunk-AI/bakeoff/issues/12#issuecomment-4494171918>

**Bastion response — 200327ZMAY26**

@gissf1 — addressing all points in order.

---

**A1: Prompt FK to task_id + task/test/test suite terminology**

Agreed — a prompt record should have a `task_id` FK. This also lets `run_model_metrics` drop the separate `task_id` field: prompt implies task.

Terminology disambiguation:

| Term | Definition |
|---|---|
| **Test suite** | A named collection of tasks run together (e.g., "baseline tier", "comparison tier"). Maps to a top-level task record with `parent_id IS NULL`. |
| **Task** | A specific evaluation type within a suite (e.g., "bash_scripting", "code_generation_python"). Corresponds to a single evaluation scenario, potentially with multiple prompt variants. |
| **Prompt** | A specific text formulation of a task. One task can have many prompt variants (for A/B testing, difficulty levels, etc.). The prompt is what gets sent to the model. |
| **Test run** | One execution of a prompt against one model on specific hardware. Produces `run_model_metrics` + `run_hardware_metrics` rows. |

"Test" as a term is overloaded — I'll avoid it in schema names and use "task", "prompt", and "run" as the canonical vocabulary.

---

**A2: File-based prompts + DB metadata — agreed**

File-based prompt content is correct: version control is the right persistence layer for prompt text. Git gives us full history, diff, and blame for free.

DB metadata stored per prompt:
- `file_path` — relative path within the repo
- `git_commit_hash` — SHA of the commit in which this prompt was last modified (deterministic unique identifier for this prompt version)
- `content_length_bytes` — byte count of the raw prompt text
- `version` — human-readable version label (e.g., "1.0", "1.1-prerelease")
- `release_date` — date this version was officially released
- `is_prerelease` — flag for prompts in testing before official release
- `task_id` FK — which task this prompt belongs to

For reproduction: given a `run_model_metrics` row, the `git_commit_hash` on its referenced `prompt` record uniquely identifies the exact prompt text used. Historical versions accessible via git. No need to store prompt text in the DB.

---

**A3: Model deduplication + versioning**

**Parent:** #8 — Additional Performance Metrics

Implement `system_hardware` and `system_software` tables as ratified in #8. These are separate entities: hardware describes the fixed physical host; software describes the runtime environment active during a specific run.

## Schemas

```sql
CREATE TABLE system_hardware (
    system_hardware_id  SERIAL PRIMARY KEY,
    system_id           UUID NOT NULL UNIQUE,  -- stable per-host UUID, generated once at first run
    publisher_id        TEXT NOT NULL,          -- submitting user/account
    cpu_model           TEXT NULLABLE,
    ram_total_gb        FLOAT NULLABLE
);

CREATE TABLE system_software (
    system_software_id  SERIAL PRIMARY KEY,
    os                  TEXT NULLABLE,           -- "Ubuntu 24.04.2 LTS"
    kernel_version      TEXT NULLABLE,           -- "6.8.0-57-generic"
    python_version      TEXT NULLABLE,           -- "3.12.3"
    gpu_driver_version  TEXT NULLABLE,           -- nvidia-smi Driver Version field
    cuda_version        TEXT NULLABLE,           -- null for ROCm/CPU-only runners
    rocm_version        TEXT NULLABLE,           -- null for CUDA runners
    runner_version      TEXT NULLABLE            -- bakeoff harness commit hash or semver
);
```

## Auto-detection requirements

### `system_hardware`

- **`system_id`** — stable UUID stored in local config (e.g., `~/.config/bakeoff/system_id`). Generated with `uuid.uuid4()` on first run; never regenerated. This identifies the physical machine across runs even when submitted by different accounts.
- **`publisher_id`** — from existing runner config (already tracked in harness)
- **`cpu_model`** — `/proc/cpuinfo` → `model name` field (Linux); `platform.processor()` fallback
- **`ram_total_gb`** — `psutil.virtual_memory().total / (1024**3)`

### `system_software`

- **`os`** — `platform.platform()` or `distro.name(pretty=True)` for Linux distributions
- **`kernel_version`** — `platform.release()`
- **`python_version`** — `platform.python_version()`
- **`gpu_driver_version`** — `nvidia-smi --query-gpu=driver_version --format=csv,noheader`
- **`cuda_version`** — `nvidia-smi` → CUDA Version field in header; or `torch.version.cuda` if torch available
- **`rocm_version`** — `rocm-smi --showversion` output; null if unavailable
- **`runner_version`** — harness `__version__` constant or `git describe --tags --always` at build time

## Deduplication

`system_hardware` deduplicates on `system_id` (unique constraint). Same machine across runs reuses the existing row.

`system_software` does **not** deduplicate — the same software environment may be recorded multiple times as a new row; `run_hardware_metrics` references whichever row was active during that run. Deduplication by full column hash is optional optimization for Phase 2.

## Acceptance criteria

- [ ] Both migrations create tables with correct types and constraints
- [ ] `system_id` UUID is generated on first run and persisted to local config; stable across subsequent runs on the same machine
- [ ] Harness auto-populates both tables at startup
- [ ] `system_hardware` upserts on `system_id` (do not create duplicate rows for the same machine)
- [ ] `system_software` inserts a new row per run (environment snapshot, not deduplicated)
- [ ] Both migrations are reversible
